note San Francisco AIDS Foundation

SFAF was in need of an overhaul of their housing subsidy management system, in the form of an entirely new application. In short, the system was used to track clients seeking a housing subsidy from the foundation and calculate their eligibility for various programs.

For this application, I worked closely with SFAF to determine the functionality required of the old application while trimming the rest, as there was a lot of existing functionality they did not need, and the application was closed source.

This was really fun to write as I was working the full stack with a completely clean slate, and was able to put together a much cleaner, lightweight app for the foundation to use.

The application included:

  • A SQL Server database for storing client information and configurable(with administrative privileges) subsidy calculation parameters.
  • A RESTful Node/Express.js backend with authentication handled via JWT and through a configurable Active Directory connection.
  • An Angular 7 frontend with login/authentication, client registration, income reporting, subsidy status reporting, subsidy calculations and historical data, an administrative panel, and client search capabilities.
  • Tested and deployed internally at SFAF on Windows Server with IIS.

    Metron Scientific Solutions

    Software Analyst and Architect

    Starting at Metron was incredibly exicting for me. Fresh out of school, with a job working on some pretty interesting problems. I was specifically brought on to work on the SAFEGARD project(see below) and over the course of my five years with Metron I went from being a junior developer on SAFEGARD, to managing the project entirely(with an occasional junior developer working for me).

    The projects I did the most appreciable work on are further detailed below:

    note SAFEGARD

    SAFEGARD is a mission level warfare analysis application - naval, air, communications, and sensor assets were modeled with a high level of fidelity to assist in the procurement analysis process for the Navy.

    SAFEGARD was my first and longest lasting assignment at Metron - I was brought on to be a junior developer on what became a three developer team, and when I arrived the software design phase had just gotten underway, and the codebase was basically a CMake file. I was elated to be joining a project with a clean slate!

    The simulation engine is written in C++, with Python(PLY) used to translate user input data, Java used as a front end and 3D results playback tool(leveraging NASA WorldWind for an OpenGL globe display and panel), and dynamically generated HTML(leveraging NextReports Engine), in conjunction with jQuery and Javascript used for result output(stored within a PostgreSQL database). With such a small team, we all worked on nearly every module within the project - an effort that took the better part of five years.

    Later, after the release of version 1.0, I was asked to produce a module which generates documentation for the vast amount of input data that SAFEGARD models require. This module leveraged the results output module already implemented(using NextReports Engine, dynamically generated HTML, jQuery, and Javascript). This effort greatly reduced analyst overhead time when designing a mission, and the effort was able to reuse most of the codebase for a similar documentation tool implemented for STORM.

    note STORM

    STORM is a campaign level warfare analysis application, that was something of the father to SAFEGARD - SAFEGARD being a less broad simulation with higher fidelity.

    Architecture and design of SAFEGARD borrowed very heavily from STORM, and after the release of SAFEGARD v1.0 I was brought on to STORM to make a few additions, leveraging my knowledge of SAFEGARD.

    The simulation engine is written in C++, and I was asked to produce high level design for a module that would model small boat swarms, and provide guidance to a junior developer on the implementation of the module.

    I was also asked to produce a documentation module, leveraging what had been written for SAFEGARD. The effort was able to reuse most of the codebase for the similar documentation tool designed for SAFEGARD.

    note ExAMS

    ExAMS is an analysis tool(written in Java using the Spring Framework).

    The tool was used to organize the littany of different sources, formats, and types of data required at a generic level for many military simulations, and architected to easily allow for development of plugins to utilize those data sources in an abritrary simulation environment.

    I was brought on to design and develop a plugin to allow for data input from real life hardware, utilizing an HLA methodology. This plugin allowed for the input data from real life hardware to be passed into an arbitrary simulation environment and be utilized at arbitrary intervals/levels of fidelity.

    Thomas Jefferson National Accelerator Facility

    Research Assistant

    [October 2010 - April 2011]

    During my senior year at CNU, I was lucky enough to land an assistantship at Jefferson Lab in Newport News, VA. So began my love of working with scientists.

    As an assistant, I was tasked with helping the group implement and test CUDA versions of various facets of their work to achieve some increased efficiency through parallelization.

    These efforts included the following:

    note Implementation of a Kalman filter

    This effort was largely unsuccessful - the algorithm, generically, does not lend itself well to parallelization. Since my task was specific to paralleization(they had a group specifically working on GPU farming processes), this exploration and initial implementation was a short effort.

    It was something of a square-peg round-hole situation, but the effort did make for a good exploration into CUDA(outside of some previous classwork I had done).

    note Implementation of a cluster finding algorithm

    This effort was much more successful than the first. The detector is made up of a large array of sensors used to record the collision. The algorithm for finding clusters of these sensors firing at the same time was quite simple, and allowed for high levels of parallelization.

    This implementation allowed the group to find clusters within their datasets around 3 times faster than the naive serial implementation.

    note Implementation of a segment finding algorithm

    Similar to the cluster finding algorithm - this algorithm for finding segments of these sensors firing at the same time(specifically looking for uncorrelated noise) was quite simple, and allowed for high levels of parallelization.

    This implementation allowed the group to find and remove noise from their datasets on the order of 1000 times faster than the naive serial implementation.
